# Phase 12 Research: Standardized Research Workflow

**Completed:** 2026-04-06
**Level:** 0 (established patterns)

## Summary

Phase 12 uses entirely established codebase patterns plus three harness-specific entry point formats already present in the repo. No new external dependencies required.

## Key Findings

### Multi-Harness Entry Point Patterns (D-06, D-07)

All three AI harnesses already have agent/skill directories in this repo:

#### GitHub Copilot (`.github/agents/`)
- Format: `.github/agents/{name}.agent.md`
- YAML frontmatter: `name`, `description`, `tools` (list)
- Body: Markdown with `<role>`, instructions, etc.
- Invocation in chat: `@{name}` mentions the agent by its YAML `name`
- Example: `.github/agents/gsd-planner.agent.md`

#### Cursor (`.cursor/skills/`)
- Format: `.cursor/skills/{name}/SKILL.md`
- YAML frontmatter: `name`, `description`
- Body: `<cursor_skill_adapter>` section with invocation rules
- Invocation: User mentions the skill name in Cursor chat
- Example: `.cursor/skills/gsd-fast/SKILL.md`

#### Claude Code (`.claude/agents/`)
- Format: `.claude/agents/{name}.md`
- YAML frontmatter: `name`, `description`, `tools`, `permissionMode`, `color`
- Body: Markdown with `<role>`, instructions
- Invocation: `/agent:{name}` or referenced by orchestrator
- Example: `.claude/agents/gsd-executor.md`

### Canonical Workflow → Thin Entry Points Architecture
Per D-06, the canonical research workflow lives at `.leanquill/workflows/research.md`. Each harness entry point is a thin wrapper that:
1. References the canonical workflow file
2. Adapts the invocation syntax to the harness convention
3. Delegates all research logic to the canonical definition

### Harness Detection for Quick-Start Button (D-10)

The extension needs to detect the active AI harness to format the pre-populated chat command. Detection strategy:

| Harness | VS Code Extension ID | Chat Command Format |
|---------|---------------------|-------------------|
| GitHub Copilot | `github.copilot-chat` | `@researcher {query}` |
| Cursor | Running inside Cursor (check `vscode.env.appName`) | `@researcher {query}` |
| Claude | `anthropic.claude-code` (if VS Code extension) or CLI-based | `Research: {query}` |
| Fallback | None detected | `Research: {query}` |

Best detection approach: Check `vscode.env.appName` first (contains "Cursor" if in Cursor IDE), then check extensions. Cursor and Copilot both support `@` agent mentions. Claude Code operates via terminal/CLI, so the "+" button fallback message is appropriate.

### VS Code Chat Participant API
- Register in `package.json` contributes.chatParticipants — NOT needed for this phase
- Decision D-05 says research is an AI skill/agent, not a LeanQuill command
- The extension does NOT register a chat participant; research runs via harness-specific agent files

### Opening Chat from Extension
- `vscode.commands.executeCommand('workbench.action.chat.open', { query: '...' })` pre-fills the chat input
- Used for the "+" button (D-09) to launch research with pre-populated invocation text
- This command is available in VS Code and Cursor (both have chat panels); fallback info message for Claude Code

### YAML Parsing Approach
- No YAML library in dependencies (only `@vscode/codicons`)
- project.yaml has a controlled, simple format generated by LeanQuill itself
- Line-based regex parsing is sufficient for extracting `schema_version` and `folders.research`
- Schema migration (v1→v2) uses string replacement on the known format

### Established Patterns to Reuse
- **TreeDataProvider**: `chapterTree.ts` — model for Research sidebar section
- **SafeFileSystem**: Allowlist model — extend with `allowPath()` for research folder
- **Config generation**: `initialize.ts` `renderProjectYaml()` — template approach for canonical workflow
- **Test isolation**: `withTempDir()` pattern from `safeFileSystem.test.ts`
- **File watching**: `vscode.workspace.createFileSystemWatcher` used throughout `extension.ts`

### Safety Boundary
- Canonical workflow lives in `.leanquill/workflows/` — already allowed by SafeFileSystem
- Research files go to `{folders.research}` — needs new allowPath entry for `.md` files only
- Harness entry points (`.github/agents/`, `.cursor/skills/`, `.claude/agents/`) — generated by extension during init alongside canonical workflow

## Architecture Decisions

1. **No YAML library**: Parse project.yaml with line-based regex (controlled format)
2. **No chat participant registration**: Research is a skill/agent file, not extension code
3. **TreeView (not webview)**: Research list is simple name+date with click-to-open — TreeDataProvider suffices
4. **Filesystem scanning (no index)**: Research documents are few; scan `{folders.research}/*.md` on demand
5. **Three harness entry points**: All generated during init — thin wrappers referencing canonical `.leanquill/workflows/research.md`
6. **Harness detection**: `vscode.env.appName` for Cursor, `vscode.extensions.getExtension` for Copilot, fallback for Claude
