# Phase 02 — UI Design Contract

> Visual and interaction contract for Phase 2 native VS Code surfaces. Generated by gsd-ui-researcher, verified later by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none — use native VS Code `TreeView` plus `WebviewView` only |
| Icon library | VS Code `ThemeIcon` / Codicons |
| Font | VS Code workbench font variables in webview; native tree inherits workbench font |

---

## Native Surface Contract

### Chapters View

- Surface type: native `TreeView` in the LeanQuill activity bar.
- Row layout: `label = chapter title`, `description = {status} · {issue count}`.
- Ordered chapters render first in strict project order from chapter-order data.
- `Not Included` renders as a collapsed group after ordered chapters and contains manuscript files absent from the chapter-order source.
- Missing files remain visible in project order with a warning icon, `(Missing)` suffix in the label, and no open command.
- Issue count remains visible as `0 issues` in Phase 2 so the scan pattern is stable before issue integration lands.

### Chapter Context Pane

- Surface type: native `WebviewView` in the LeanQuill activity bar.
- Layout: single-column summary panel with chapter title, status block, open-issue block, and helper text.
- Notes section is hidden for this phase.
- The pane keeps showing the last known chapter when the active editor is not a known chapter.
- The pane uses static HTML and CSS only. Scripts remain disabled.

### Status Presentation

- Statuses use the canonical sequence from requirements: `planning`, `not-started`, `drafting`, `draft-complete`, `editing`, `review-pending`, `final`.
- Represent status primarily with icon + text, not color-only encoding.
- Use this icon mapping consistently across tree rows, quick pick entries, and the context pane:

| Status | Icon |
|--------|------|
| planning | `circle-outline` |
| not-started | `dash` |
| drafting | `edit` |
| draft-complete | `pass` |
| editing | `pencil` |
| review-pending | `clock` |
| final | `verified` |

## Interaction Contract

| Interaction | Contract |
|-------------|----------|
| Open chapter | Single click previews the manuscript in the editor. Double click pins the tab. |
| Update status from tree | Right-click chapter row, choose `Update Chapter Status`, then select from a quick pick showing all 7 statuses in canonical order. |
| Update status from context pane | Primary action uses the same quick-pick flow as the tree so there is one status-edit path. |
| Auto-refresh | Refresh the context pane when the active editor changes to a known chapter and after a successful status update. |
| Unknown active file | Keep the last known chapter in the context pane instead of clearing to empty. |
| Missing chapter row | Show warning state only; do not attempt open or auto-repair from the row. |
| Error recovery | On invalid stored status, warn once, fall back to `not-started`, and continue rendering. |

---

## Layout Contract

- Chapters tree uses the native VS Code compact density. Do not simulate custom cards or badge pills inside the tree.
- Chapter Context Pane uses one primary summary card and one secondary metadata card stacked vertically.
- Each context card uses 16px internal padding, 8px internal row gaps, and 24px separation between cards.
- Chapter title stays on one line when possible and truncates with ellipsis before wrapping into a multi-line block.
- Metadata rows use label-left/value-right alignment for fast scanning.
- Primary status action sits directly below the status value, not in a floating toolbar.

---

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline separators |
| sm | 8px | Compact row spacing, small status/meta gaps |
| md | 16px | Default card padding, default webview spacing |
| lg | 24px | Separation between pane sections |
| xl | 32px | Top-level pane breathing room |
| 2xl | 48px | Empty-state vertical spacing |
| 3xl | 64px | Major top/bottom margins in the webview |

Exceptions: native tree row height, indentation, disclosure spacing, and selection chrome stay under VS Code control; webview interactive controls may use a 28px minimum height.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 13px | 400 | 1.5 |
| Label | 12px | 600 | 1.3 |
| Heading | 16px | 600 | 1.25 |
| Display | 20px | 600 | 1.2 |

Rules:

- Use only weights 400 and 600.
- Tree typography stays native to VS Code; these sizes apply to webview content only.
- Label style is reserved for metadata keys, helper labels, and section captions.
- Display style is reserved for the chapter title in the context pane only.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `var(--vscode-sideBar-background)` | View background, webview root, inactive surface fill |
| Secondary (30%) | `var(--vscode-editorWidget-background)` | Summary cards, grouped metadata blocks, empty/error panels |
| Accent (10%) | `var(--vscode-button-background)` | `Update Chapter Status` action and any single primary action affordance in the pane |
| Destructive | `var(--vscode-errorForeground)` | Destructive actions only |

Accent reserved for: the `Update Chapter Status` action in the context pane, inline retry/open-state links inside empty or error panels, and no other elements.

Additional rules:

- Do not hardcode hex colors; all webview colors must be derived from VS Code theme tokens.
- Do not assign a unique color to every chapter status. Status identity comes from icon + label first.
- Missing-file warnings use VS Code warning semantics, not a custom palette.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Update Chapter Status |
| Empty state heading | No chapter selected |
| Empty state body | Open a chapter from the Chapters view to inspect its status and issue context. LeanQuill keeps the last known chapter visible when you switch to non-chapter files. |
| Error state | LeanQuill could not load chapter context. Reopen the chapter, then retry the status action. If project state is missing, run LeanQuill: Initialize. |
| Destructive confirmation | None in Phase 2: no destructive actions are in scope. |

Microcopy rules:

- Use `chapter`, not `document` or `file`, in user-facing action labels.
- Use sentence case for headings, buttons, quick-pick placeholders, and helper text.
- Missing-file placeholder rows should read as actionable warnings, for example: `Chapter file missing from manuscript/`.
- Invalid-status warnings should name the stored value and the fallback, for example: `Unknown chapter status "stalled". LeanQuill is showing not-started instead.`

---

## Empty, Warning, and Error States

| State | Contract |
|-------|----------|
| Initial pane empty state | Show the empty-state heading/body above with no placeholder illustration. |
| Non-chapter editor active | Keep previous chapter context visible and add a subdued helper line: `Showing last active chapter context.` |
| Missing manuscript target | Show warning icon + missing label in tree and a warning panel in the context pane if that chapter becomes active through stored state. |
| Invalid stored status | Show a warning message, coerce to `not-started`, and continue rendering. |
| Issue integration pending | Show `0 issues` plainly; do not add disabled issue controls or teaser buttons in Phase 2. |

---

## Accessibility and Safety

- Every state must be understandable without color alone.
- Webview markup must include a strict CSP with `default-src 'none'` and inline styles only.
- Webview scripts remain disabled for this phase.
- Keyboard flow must match native VS Code behavior: tree navigation via arrow keys, context menu accessible from keyboard, and quick pick fully keyboard-operable.
- Do not invent hover-only actions; all core actions must be visible through the context menu or primary pane action.
